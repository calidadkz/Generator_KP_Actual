## Концепция

Агент — это не просто AI который отвечает на вопросы. Это интерактивный ассистент-администратор внутри приложения.

**Аналогия:** как Claude Code помогает разработчику писать приложение, так Агент помогает руководителю *администрировать* это приложение.

**Три ключевых отличия от обычного AI:**
1. **Память** — помнит весь контекст разговора и знает базу данных
2. **Действия** — может писать, изменять, удалять данные в Firestore после согласования
3. **Многошаговость** — обсудили → решили → выполнил → подтвердил

## Процесс: обычный рабочий сеанс

```
Руководитель открывает приложение →
Колокольчик 🔔(3) → "Аудит понедельника готов: найдено 2 дубля МП"

Открывает чат агента:
"Покажи дубли"
Агент: [поиск в Firestore] → "МП-83 и МП-91 похожи на 80%.
  МП-83: [текст про чиллер CW-5200, общая формулировка]
  МП-91: [текст про чиллер CW-5200, для акрила 10мм]
  Предлагаю объединить. Вот новая формулировка: [...]"

"Добавь фразу про гарантию трубки"
Агент: "Добавил: [...обновлённый текст с фразой про гарантию...]
  Сохранить в базу?"

"Да"
Агент: [запись в Firestore, soft-delete МП-91] → "Готово.
  Журнал: МП-83 обновлена, МП-91 помечена как удалённая (можно откатить)."
```

## Архитектура: три слоя

### Слой 1 — Инструменты (в коде, фиксированы)
«Руки» агента — базовые операции с базой данных. Меняются только разработчиком при добавлении новых коллекций.

```
Библиотека:
  search_knowledge(query)           — RAG поиск по всем коллекциям
  create_micro_presentation(...)    — создать МП
  update_micro_presentation(id, {}) — обновить МП
  delete_micro_presentation(id)     — мягкое удаление МП

Скрипт:
  get_script_nodes()
  update_script_node(id, {})
  create_script_node(...)

Диалоги:
  get_dialogues(filters)
  search_dialogues(query)

Рутины и журнал:
  create_routine(...)
  run_routine(id)
  log_action(...)                   — запись в журнал аудита
  generate_report(type, params)
```

### Слой 2 — Системный промт (настраивается руководителем)
«Правила и характер» агента. Меняется в интерфейсе без разработчика.

Пример содержимого:
```
Ты — Knowledge Architect системы CALIDAD.kz.
Ты знаешь структуру всех коллекций Firestore.

ПРАВИЛА ПОВЕДЕНИЯ:
- Перед созданием МП всегда проверяй дубли
- Для удаления/массовых изменений требуй явного подтверждения
- При оценке менеджера использовать активный чек-лист

ТЕКУЩИЕ ПРИОРИТЕТЫ:
- Наполнение библиотеки по волоконным лазерам
- [руководитель меняет список в интерфейсе]
```

### Слой 3 — Рутины (настраиваются в интерфейсе или через чат)
«Обязанности» агента — сохранённые цепочки действий с расписанием.

**Создание через чат:**
```
"Мне нужно каждый понедельник проверять дубли в библиотеке"
Агент: "Создаю рутину 'Аудит дублей':
  Шаги: 1) поиск похожих МП  2) поиск неиспользуемых МП  3) отчёт
  Расписание: каждый понедельник в 02:00
  [Сохранить в список рутин]"
```

**Создание через форму:** `+` → заполнить поля → сохранить

## RAG — умный поиск

Агент не «знает» базу наизусть — он умеет искать.

**Два типа поиска:**

| Тип | Когда работает | Пример |
|-----|----------------|--------|
| Точный | Конкретные термины, модели, цифры | "380В", "CW-5200" |
| Семантический | Концепции и смыслы | "объяснить охлаждение" → находит всё про чиллеры и перегрев |

**Техническая реализация:** эмбеддинги через Gemini Embedding API + Firestore vector search. Уже доступно в текущем стеке (Firebase + GCP).

## Память между сессиями

**НЕ хранить** полную историю чата — дорого и избыточно.  
**Хранить** структурированные резюме завершённых задач:

```json
{
  "date": "2026-05-20",
  "task": "Аудит дублей МП",
  "summary": "Удалено 5, объединены 3 группы. 2 на просмотре (mp-47, mp-83)",
  "openItems": ["Проверить mp-47", "Проверить mp-83"],
  "tags": ["library", "duplicates"]
}
```

При старте сессии: загружаем последние 10 резюме → агент знает контекст без дорогой истории.

Коллекция Firestore: `agent_memory`

## Журнал действий + откат

Коллекция Firestore: `agent_audit_log`

```json
{
  "timestamp": "2026-05-20T14:32:00",
  "action": "update_micro_presentation",
  "collection": "micro_presentations",
  "documentId": "mp-83",
  "triggeredBy": "agent",
  "context": "обсуждали дубли МП про чиллер",
  "before": { "title": "...", "content": "старый текст" },
  "after":  { "title": "...", "content": "новый текст с гарантией" }
}
```

**Откат:** одна кнопка → документ восстанавливается из поля `before`.

### Три уровня риска действий

| Уровень | Примеры | Поведение агента |
|---------|---------|-----------------|
| 🟢 Зелёный | Добавить тег, создать черновик МП | Выполняет сразу |
| 🟡 Жёлтый | Изменить текст МП, обновить этап скрипта | Показывает предпросмотр → ждёт подтверждения |
| 🔴 Красный | Удалить МП, массовые изменения | Явное подтверждение + soft delete (можно откатить) |

## Рутины — интерфейс «Обязанности администратора»

```
┌──────────────────────┬─────────────┬──────────┬───────────────┐
│ Название             │ Расписание  │ Статус   │ Последний     │
├──────────────────────┼─────────────┼──────────┼───────────────┤
│ Аудит дублей МП      │ Пн 02:00    │ ✅ Вкл   │ сегодня 02:00 │
│ Отчёт по менеджерам  │ Пт 17:00    │ ✅ Вкл   │ 16 мая        │
│ Еженедельный Gap     │ Вс 03:00    │ ✅ Вкл   │ 18 мая        │
│ Дайджест задач       │ Вручную     │ ⏸ Выкл  │ 10 мая        │
└──────────────────────┴─────────────┴──────────┴───────────────┘
```

При клике на рутину: описание + история 10 запусков + `[Запустить сейчас]` + `[Редактировать]`.

**Архитектура расписания (гибрид):**
- Cloud Scheduler (GCP, уже есть в проекте) → рутина запускается ночью → результат в Firestore
- При открытии приложения → колокольчик 🔔 показывает новые результаты
- Критичные рутины → Cloud Scheduler. Остальные → при открытии приложения.

## Обработка дублей МП — подход «Кластеризация»

Агент группирует похожие МП и показывает предложение:

```
⚠️ Найдено 3 похожих МП про "Чиллер CW-5200"
  МП-47: [общая формулировка] — использована 12 раз
  МП-83: [для акрила 10мм]   — использована 8 раз
  МП-91: [про перегрев]      — использована 3 раза

  Агент предлагает: объединить МП-47 в МП-83, МП-91 оставить.
  Новая формулировка МП-83: [...]

  [Принять] [Редактировать] [Оставить все]
```

Переформулирует сохраняя общую стилистику утверждённых ответов. Руководитель может исправить и добавить.

## Агент → задачи для разработки (Claude Code)

```
Руководитель замечает нужную доработку
→ Описывает агенту: "В тренажёре нет сценариев для Дожима"
→ Агент формулирует техническую задачу:
   "ЗАДАЧА: Добавить тип сценария 'Дожим' в тренажёр
    Затрагивает: training_scenarios (Firestore), TrainerInterface.tsx
    Тесты: проверить загрузку сценария, возражения для этапа Дожима
    [Скопировать для Claude Code]"
→ Руководитель копирует в Claude Code → реализует
→ Агент генерирует чек-лист тестирования
```

Рутина «Еженедельный дайджест задач»: компилирует накопленные улучшения в единый документ-спецификацию.

## Тренажёр (режим агента для менеджеров)

> Подробное описание: [[10_trainer]]

**Ключевые принципы:**
- Отдельный интерфейс — менеджер не видит admin-агента
- Настраивается через admin-агент (руководитель создаёт сценарии)
- Агент играет роль клиента на основе реальных портретов из диалогов

**Создание сценария через admin-агент:**
```
"Создай сценарии для волоконного лазера"
→ Агент находит 15 диалогов → кластеризует по 3 типам клиентов
→ Генерирует 3 черновика досье → руководитель правит → тест-прогон
```

**Два режима для менеджера:**
1. **Свободная тренировка** — выбирает сценарий сам из доступных
2. **Наряд-задание** — руководитель назначает N сценариев с дедлайном → агент формирует отчёт

**Шестерёнка в тренажёре = выявление пробелов скрипта:**
```
Менеджер не знает ответ на вопрос клиента →
нажимает шестерёнку → "Не знал ответить про заземление 380В"
→ падает руководителю → добавляет МП в скрипт
→ следующая тренировка уже с этим знанием
```
