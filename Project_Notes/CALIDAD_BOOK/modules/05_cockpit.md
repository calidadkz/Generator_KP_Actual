## Концепция

Кокпит — инструмент «живого» звонка. Менеджер ведёт разговор с клиентом и одновременно идёт по скрипту, получает актуальные МП, фиксирует квалификационные данные.

**Ключевой принцип:** скрипт адаптируется к ответам клиента в реальном времени — не просто ведёт по этапам, а учитывает что клиент сказал и показывает именно те МП которые нужны прямо сейчас.

## Концепция «Задача» (переосмысление мульти-изделия)

**Единица навигации = Задача** (не «изделие»).

Задача = что нужно сделать + каким типом станка это решается.

```
Примеры:
Изделие «деревянная вывеска» → ДВЕ задачи:
  Задача 1: Вырезать форму → CO2 лазер или фрезер
  Задача 2: Нанести принт → DTF или UV-принтер
  
«Резать фанеру 6мм и 20мм» → ОДНА задача:
  Задача 1: Резка дерева → CO2 лазер (6мм) или фрезер (20мм)
  (разные рекомендации но один тип задачи)
  
«Резать акрил + резать металл» → ДВЕ задачи:
  Задача 1: Резка акрила → CO2 лазер
  Задача 2: Резка металла → Волоконный лазер
```

**UI:** вкладки задач в верхней части Кокпита. Клик по вкладке → весь скрипт + правая панель обновляются под эту задачу.

## Слоты квалификации («Живая сессия»)

Слот = конкретный параметр который нужно выяснить у клиента.

```
Слоты для задачи «Резка CO2»:
  ✅ material    = "акрил"      (выяснили, скрипт адаптировался)
  ✅ thickness   = "10мм"       (выяснили)
  ⏳ volume      = ?            (клиент сказал «уточню и напишу»)
  ✅ budget_range = "до 15 млн" (выяснили)
```

**Принцип адаптации:** правая панель обновляется по тому что УЖЕ известно, не ждёт заполнения всех слотов.

```
Известен материал: акрил → показываются МП про CO2 для акрила
После заполнения толщины 10мм → добавляется МП «Чиллер для акрила 10мм»
Неизвестен объём → МП про производительность не показываются
```

### Ожидающие слоты (⏳)

Когда клиент не знает ответ прямо сейчас:
- Менеджер помечает слот как ⏳ «жду данные»
- Разговор продолжается без этого параметра
- Позже: клиент пишет в WhatsApp → менеджер открывает сессию → дозаполняет
- Правая панель обновляется автоматически

### Статусы сессии

| Статус | Значение |
|--------|---------|
| 🟢 Активная | Звонок идёт сейчас |
| 🟡 Пауза | Ждём данные от клиента |
| ✅ Завершена | Вся квалификация собрана |

Менеджер видит список своих незакрытых сессий — легко вернуться.

## Авто-адаптация скрипта

При выборе некоторых значений скрипт автоматически меняет поведение.

**Примеры:**
```
[Материал: ПВХ] → предупреждение: «Хлор в ПВХ разрушает оптику CO2 лазера»
                → показываются МП «Почему не лазер для ПВХ» + рекомендация фрезера

[Материал: фанера] + [Толщина: до 10мм] → МП «CO2 vs Фрезер для тонкого дерева»
[Материал: фанера] + [Толщина: 15мм+]  → МП «Почему фрезер для толстой фанеры»
```

**Правила авто-адаптации:** агент выводит из диалогов → руководитель подтверждает → хранятся в ScriptNode как условия:
```json
{ "if": { "material": "ПВХ" }, "action": "show_warning", "warningText": "..." }
```

## Резюмирующая Вилка

Не отдельный функционал — результат правильно заполненного скрипта с МП и условиями.

**Три формы Вилки:**

**Форма А — Опции одного типа станка:**
```
CO2 1390 базовый: X тенге (80W, без чиллера)
CO2 1390 стандарт: Y тенге (100W, с чиллером)
CO2 1390 расширенный: Z тенге (150W, с ротором)
```

**Форма Б — Разные типы под разные подзадачи:**
```
Если приоритет 6мм фанеры: CO2 лазер 80W → X тенге
Если приоритет 20мм фанеры: Фрезер 1500W → Y тенге
Если обе задачи: CO2 80W + Фрезер → X+Y тенге
```

**Форма В — Нет Вилки (один вариант):**
```
Гравировка металла → Волоконный лазер 20W
(единственный правильный выбор — Вилка не нужна)
```

**Принцип:** не давать слишком много вариантов. Скрипт отсекает лишнее по бюджету/приоритету. Менеджер зачитывает финальные 2-3 варианта для подтверждения: «Верно?»

## Нестандартная задача

**Когда использовать:** задача принципиально за пределами ассортимента CALIDAD (например: производственная линия для манжетов, промышленный конвейер).

**Важно:** «не нашёл нужный станок» — НЕ нестандартная задача. Фанеру можно резать и CO2, и фрезером — это нормальная ситуация с выбором.

**Два триггера:**
1. Кнопка `[⚠️ Нестандартная задача]` — всегда видна, менеджер нажимает осознанно
2. В конце ветки квалификации — скрипт спрашивает: «Задача вписывается в стандартный ассортимент?»

**Режим сбора информации:**
```
Скрипт переходит в режим «Сбор для инженера»:
  — Опишите изделие которое нужно производить?
  — Какой материал?
  — Какие операции нужно выполнять на станке?
  — Приблизительный объём/количество?
  — Желаемые сроки?

Финал: сводка → договориться о созвоне с инженером (Евгением)
```

**Цикл оцифровки нестандартного:**
```
Менеджер собирает данные
→ Евгений проводит консультацию с клиентом (разговор записывается)
→ Разговор Евгения загружается в систему
→ Агент оцифровывает: создаёт МП, добавляет условия в скрипт
→ Задача перестаёт быть «нестандартной» для следующих менеджеров
```

## Заполнение скрипта через агент-чат

Основной канал наполнения экспертными знаниями — разговор руководителя с агентом.

```
Руководитель в агент-чате:
«Для фанеры 6мм лучше CO2, для 20мм — фрезер. Потому что лазер дороже
в обслуживании для толстого материала, а фрезер стабильнее»

Агент задаёт уточняющие вопросы:
«Пороговое значение — до 10мм лазер, от 10мм фрезер? Или другая граница?»

«Добавить МП "CO2 vs Фрезер для дерева" в этап Квалификации?
  Условия показа: [Материал: дерево] + [Толщина: ?]
  [Подтвердить] [Редактировать порог]»
```

Руководитель объясняет методику так же как обучает менеджеров — агент оцифровывает это в МП и условия скрипта.

При переходе к скрипту Дожима — сводка квалификации остаётся доступна.

**Сейчас:** сессия хранится в Firestore. Менеджер выбирает из списка последних сессий.


## Методика бюджета и компромисса

Бюджет — не жёсткое ограничение. Скрипт помогает найти баланс между задачей и возможностями.

**Три сценария:**

```
А) Бюджет ≈ решение → стандартная Вилка

Б) Бюджет чуть меньше → Компромисс + план апгрейда:
   «Рабочее поле 1500мм сразу, чиллер — позже.
    Пока базовая помпа — будете контролировать температуру сами,
    но потом добавите. Станок — конструктор.»

В) Бюджет сильно меньше → Честный отказ или переориентация:
   Пример: 1.5 млн на резку металла (станок 10 млн) → нет.
   Можно предложить: волоконный маркер для гравировки (3 млн),
   но резать металл при таком бюджете нельзя.
```

**Принцип «конструктора»:** базовая модель предусматривает апгрейд. Менеджер выравнивает ожидания: «Вашу задачу понимаю, учитывая бюджет — можно сделать так. В будущем легко дополнить.» Часто после этого клиент либо добавляет бюджет, либо берёт рассрочку.

**Бюджет в сессии:** один глобальный текстовый слот на всю сессию (не на отдельную задачу). Фиксируется свободным текстом. Скрипт учитывает через МП с методиками компромисса.

## Мульти-задача — рекомендованная организация

**Рекомендованный подход (Guided Sequencing + Progress):**

```
[Задача 1: Резка ●●●○○] [Задача 2: Печать ○○○○○] [+ Задача]

Менеджер говорит клиенту: «Давайте сначала разберём лазер,
потом перейдём к принтеру»

→ Работает по Задаче 1
→ Когда ключевые слоты заполнены, скрипт:
  «Данные по лазеру собраны. Переходим к Задаче 2?»
  [Да, переходим] [Нет, продолжу]
→ При переключении показывает резюме Задачи 1 (1 строка):
  «Лазер CO2: акрил 10мм, серийный, бюджет ~10 млн»
```

**Вкладки задач видны всегда** — менеджер контролирует порядок. Скрипт помогает, не диктует.

### Группа 1: Когнитивная нагрузка

| # | Риск | Решение |
|---|------|---------|
| 1 | 5 параллельных потоков (слушать + кликать + читать + ERP + вкладки) | Тестировать реальными хронометражом. Если >3-4 клика/мин — убирать функции |
| 2 | Потеря места при переключении задач | Прогресс-индикатор + резюме одной строкой при переключении |

### Группа 2: Качество скрипта

| # | Риск | Решение |
|---|------|---------|
| 3 | Пустая правая панель (МП ещё не добавлены для этой комбинации) | Показывать общие советы этапа из `tips[]` пока МП не добавлены |

| 5 | Слоты субъективны («серийный» = 50 или 1000 в месяц?) | Для критичных параметров — порядок цифр (10 шт vs 1000 шт), не категория |

### Группа 3: Управление сессиями

| # | Риск | Решение |
|---|------|---------|
| 6 | Накопление брошенных «ожидающих» сессий | Алерт: «Сессия 14 дней без обновлений → закрыть?» |
| 7 | Дубликаты сессий одного клиента | При создании: проверка по телефону + предложение продолжить старую |

### Группа 4: Мульти-задача

| # | Риск | Решение |
|---|------|---------|
| 8 | Две задачи — один станок (менеджер не знает) | Скрипт проверяет перекрытие: «Задачи 1 и 2 решаются одним CO2. Объединить?» |
| 9 | Бюджет не учитывает несколько задач | Бюджет = один текстовый слот на всю сессию, учитывается при Вилке |

### Группа 5: Нестандартная задача

| # | Риск | Решение |
|---|------|---------|
| 10 | Новый менеджер не знает что «нестандартно» | Кнопка требует краткого обоснования (1-2 слова) + тренажёр с кейсами |
| 11 | Разговор инженера не загружен → знание теряется | Автоматически создаётся «задание на оцифровку» в очереди руководителя |
| 12 | Запись не сработала | Евгений диктует резюме в агент-чат → агент структурирует → добавляет в базу |

### Группа 6: Правила адаптации

| # | Риск | Решение |
|---|------|---------|
| 13 | Правило устарело (новый станок умеет то что раньше запрещалось) | Агент отслеживает: изменился тип станка → предлагает пересмотреть связанные правила |
| 14 | «Белый шум» — слишком много предупреждений | Только критичные риски — предупреждения. Технические нюансы — в МП |

### Группа 7: Главный риск

**Скрипт ценен только при регулярном использовании.** Первые 2-3 недели критичны. Если создаёт трение — менеджеры бросают.

Рекомендация: запустить с 1-2 простыми типами задач (CO2 лазер для дерева/акрила), потом добавлять. Не пытаться заполнить всё сразу.

## Пре-квалификационная маршрутизация

Первый вопрос скрипта: «Расскажите о вашей задаче» (+ дополнительные маршрутизаторы).

После ответа скрипт предлагает маршрут, менеджер корректирует:

```
[Знает задачу, конкретный запрос]
  → Стандартная квалификация

[Не знает что хочет, «просто смотрит»]
  → Образовательный flow:
    Отправить 2-3 видео возможностей
    «Посмотрите, перезвоним через 10 минут» — держать горячей
    После просмотра: узнать что резонировало → квалифицировать под это

[Есть ТЗ/тендер, прислал спецификацию]
  → Flow анализа ТЗ:
    Скрипт помогает расшифровать противоречия в документе
    (часто: 3 контроллера в одном станке, несовместимые требования)

[Демозал/видеовстреча/оффлайн]
  → Пока вне скрипта. Фиксировать для будущего.
```

**Количество маршрутизирующих вопросов:** начать с нескольких, после оцифровки диалогов — оставить самый эффективный.

## Версионирование скрипта

**Принцип «сначала добавляй, не переименовывай»:**

| Тип изменения | Применяется к открытым сессиям | Создаёт v2 |
|---------------|-------------------------------|------------|
| + Новый этап в конце | ✅ Автоматически | ❌ |
| + Новая МП | ✅ Автоматически | ❌ |
| + Новый слот (пустой) | ✅ Автоматически | ❌ |
| Изменение текста/советов | ✅ Автоматически | ❌ |
| Переименование слота/этапа | ❌ Нельзя применить | ✅ |
| Удаление/перестановка этапов | ❌ Нельзя применить | ✅ |

При структурных изменениях: создаётся v2. Открытые сессии остаются на v1. Менеджер видит уведомление: «Доступна новая версия скрипта [Перейти] [Завершить на текущей]».

## Аналитика скрипта (для руководителя)

```
СКРИПТ CO2 ЛАЗЕР — Май 2026

Время на этапах (среднее):
  Открытие:                 1.2 мин ✅
  Квалификация материала:   4.1 мин ✅
  Просчёт:                 23.5 мин ⚠️ (долго — проверить)
  Возражения:               2.0 мин — пропускается в 67% сессий ⚠️

Слоты чаще всего в ⏳:
  Объём производства: 41% сессий не заполнен
  Бюджет: 29% сессий не заполнен

МП с низкой открываемостью:
  «Чиллер CW-5200»: открыта в 8% случаев (пересмотреть или убрать)

Шестерёнки по этапам:
  Просчёт: 12 заметок → «как считать с чиллером» (4), «ERP не открылась» (3)
```

**Важно:** время замеряется только для «живых» сессий (статус Активная во время звонка). Сессии заполненные постфактум в аналитику времени не включаются.

## Скрипт для переписки — режим «Справочник»

Скрипт не адаптирован к асинхронной переписке (WhatsApp). Вместо последовательных этапов — режим быстрого поиска:

```
Менеджер ведёт переписку с клиентом →
Открывает Справочник →
  🔍 «чиллер аргументы» → МП + технический блок
  🔍 «380В объяснение» → атом из библиотеки
  🔍 «видео резка кожи» → ссылка из Академии

Параллельно: открывает сессию и вручную заполняет слоты
по мере того как клиент отвечает (в течение дня)
```

## ERP в Кокпите — встроенное окно (не отдельная вкладка)

**Принципиальное решение (из Q&A):** ERP открывается прямо внутри Кокпита, на этапе Просчёт — не в отдельном браузерном окне и не в отдельной вкладке приложения.

```
Скрипт (левая панель) + ERP iFrame (нижняя или правая зона) — одновременно на экране.
Менеджер не переключает контекст — скрипт и ERP видны вместе.
```

**Почему важно:** при переключении в другое окно теряется фокус звонка. Менеджер продолжает разговор с клиентом, держа перед собой и скрипт, и конфигуратор.

**Реализация:** iFrame с ERP REST API → авторизация через токен сессии.
Подробно: [[modules/13_erp_integration]]

## Динамические переменные ERP в текстах МП

МП могут содержать ссылки на конкретные позиции ERP. При показе в Кокпите переменная заменяется актуальной ценой из ERP.

**Синтаксис:**
```
{ERP_Option_Price_ID_55}  →  2 450 000 ₸
{ERP_Model_Name_ID_55}    →  CO2 1390 Standard
{ERP_Stock_ID_55}         →  В наличии (2 шт)
```

**Пример МП с переменной:**
```
«С чиллером CW-5200 ваша трубка проживёт в 3 раза дольше.
Добавить к комплектации: {ERP_Option_Price_ID_55}»
```

При отображении в Кокпите:
```
«С чиллером CW-5200 ваша трубка проживёт в 3 раза дольше.
Добавить к комплектации: 145 000 ₸»
```

**Что это даёт:** цены в МП всегда актуальны. Руководитель прописывает аргумент один раз — цена обновляется автоматически из ERP.

**К реализации:** нужна синхронизация ERP → Firestore по расписанию (раз в N часов).

## Полуавтоматический захват (после звонка)

Менеджер проводит звонок без Кокпита (не успел открыть, клиент был неожиданным) — или Кокпит был открыт но слоты не заполнены полностью.

**После звонка:**
```
Менеджер нажимает [Разобрать звонок]
→ Загружает транскрипт (или диктует резюме голосом)
→ Агент анализирует текст и предлагает заполнить слоты:
    «Обнаружено:
      Материал: "акрил" (6 упоминаний)  [✅ подтвердить]
      Толщина: "10мм" (2 упоминания)     [✅ подтвердить]
      Бюджет: не упомянут               [✏️ ввести вручную]
      Объём: "штук двести в месяц"       [✅ подтвердить]»
→ Менеджер подтверждает/корректирует → сессия заполнена
```

**Что это даёт:** даже если Кокпит не использовался во время звонка, данные не теряются. База знаний пополняется из всех звонков.
