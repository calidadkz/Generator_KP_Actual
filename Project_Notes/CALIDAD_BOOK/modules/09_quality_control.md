# 09. Контроль качества звонков

**Статус:** 💡 Запланирован (концепция проработана из реальных диалогов)
**Роли:** Руководитель (настраивает чек-лист, видит отчёты), Менеджер (видит свой разбор)
**Связи:** [[modules/07_agent]] [[modules/10_trainer]] [[modules/11_analytics]] [[modules/05_cockpit]]

---

## Для чего нужен

Каждый реальный звонок становится обучающим материалом. Агент оценивает запись по чек-листу → руководитель видит конкретные слабые места → менеджер получает точечную обратную связь вместо общих слов.

**Принцип:** одна метрика для реальных звонков и тренировок → данные сопоставимы, прогресс виден.

---

## Процесс

```
Звонок состоялся
  ↓
Запись в Битрикс24 → выгрузка → транскрибация (STT или вручную)
  ↓
Загрузка транскрипта как диалог (AdminDialogues)
  ↓
Агент оценивает по активному чек-листу руководителя
  → Для каждого критерия: ✅ выполнено / ⚠️ частично / ❌ не выполнено + цитата
  ↓
Отчёт:
  Открытие: 9/10 ✅  Квалификация: 6/10 ⚠️  Возражения: 4/10 ❌
  Системная ошибка: «не уточнял объём производства»
  ↓
Руководитель:
  — Видит сводку по всем менеджерам
  — Даёт точечную обратную связь: «Иван, работай над Возражениями»
  — Назначает наряд в Тренажёре: «пройди сценарий CO2 Эконом до пятницы»
  ↓
Менеджер → Тренажёр (отрабатывает именно слабый этап)
  ↓
Прогресс в аналитике: сравнение реальных звонков vs тренировки по неделям
```

## Чек-лист (извлечён из реальных тренинговых записей)

Это живой документ — формируется из образцовых диалогов Евгения, уточняется в процессе.
Руководитель настраивает критерии в интерфейсе, вес каждого критерия задаётся вручную.

### Блок 1: Позиция эксперта
**Почему важно:** Если менеджер находится «снизу» — просит, ждёт одобрения, заискивает — клиент ведёт себя как капризный ребёнок. Позиция эксперта = клиент всё равно купит, даже если будет недоволен.

| Критерий | Индикаторы нарушения |
|----------|---------------------|
| Не занимает позицию «просящего» | «Пожалуйста, можем мы созвониться?», «Если вы не против...» |
| Не ждёт одобрения клиента | Пауза после каждого вопроса с ожиданием похвалы |
| Голос уверенный, не дрожащий | Слышна неуверенность, оправдания |
| Не начинает с доброжелательности — дожидается её от клиента | Первым проявляет тепло до того как клиент его проявил |
| Фактами, а не мнением | «Я думаю...» вместо «Головка XP600 — срок службы меньше, I3200 дороже, это факт» |

### Блок 2: Открытые вопросы и квалификация
**Почему важно:** Закрытые вопросы дают «да/нет», открытые — поток информации. Задача — «завести пропеллер» и больше слушать.

| Критерий | Хорошо | Плохо |
|----------|--------|-------|
| Входной вопрос открытый | «Для каких задач рассматриваете станок?» | «Вам фрезерный нужен?» |
| Задаёт по материалу | «Какие материалы планируете обрабатывать?» | — |
| Задаёт по толщине | «Какая максимальная толщина?» | — |
| Задаёт по рабочему полю | Указывает варианты, спрашивает что ближе | — |
| Задаёт по конечным изделиям | «Какие конечные изделия планируете выпускать?» | — |
| Задаёт по объёму производства | «Это для производства или хобби?» | — |
| Задаёт по бюджету | Задаёт в конце, когда доверие установлено | Задаёт первым вопросом |
| Не перебивает | Дожидается конца ответа | Говорит поверх клиента |

### Блок 3: Заполнение пауз — дисциплина молчания
**Почему важно:** Если менеджер заполняет паузы сам — клиент замолкает. Пауза должна давить на клиента, не на менеджера.

| Критерий | Описание |
|----------|---------|
| Пауза после вопроса | Менеджер ждёт — не торопит, не переформулирует сразу |
| Не «гонит коней» | «У вас никто не стоит за спиной» — дать время клиенту думать |
| Слушает весь ответ | Не перебивает в середине фразы клиента |

### Блок 4: «Программирование» (установка повестки)
**Почему важно:** Сказать клиенту в начале что произойдёт — снимает сопротивление. Клиент не сопротивляется вопросам если знает зачем они.

Пример формулировки:
> «Я задам несколько вопросов о ваших задачах — мне нужно понять комплектацию. Это займёт 5 минут, после этого я смогу вам подобрать вариант.»

| Критерий | Описание |
|----------|---------|
| Задаёт повестку в начале | Объяснил зачем вопросы, что будет в итоге |
| При сопротивлении повторяет повестку | Возвращается к «для того чтобы подобрать вам правильную комплектацию» |

### Блок 5: Квалификация ДО презентации
**Почему важно:** Если начать презентацию до квалификации — клиент замолкает. «Когда вы дадите ему понимание — он заткнётся». Нужно вытащить всё сначала.

| Критерий | Описание |
|----------|---------|
| Не делает презентацию до конца квалификации | Нет «у нас есть отличный вариант» до сбора всех слотов |
| Пояснения — только по необходимости | Краткое объяснение только когда клиент не понимает термин |
| Экспертная вставка — с возвратом к квалификации | «Кстати, важный момент [кратко]... А теперь скажите, ...» |

### Блок 6: Резюмирование перед переходом
**Почему важно:** Резюме показывает клиенту что его услышали и создаёт «точку согласия» перед КП.

| Критерий | Описание |
|----------|---------|
| Резюмирует слоты | «Итак: рабочее поле 1300×2500, сеть 220, шпиндель 3.2 кВт, водяное охлаждение. Верно?» |
| Предлагает 2-3 варианта | «Покажу базовый и чуть дороже — со стабильным контроллером» |
| Не больше 3 вариантов | Больше вариантов = паралич выбора |

### Блок 7: Договорённость о следующем шаге
**Почему важно:** Без конкретного следующего шага клиент «думает» бесконечно.

| Критерий | Описание |
|----------|---------|
| Называет конкретный следующий шаг | «Пришлю КП через час, созвонимся завтра в 11» |
| Не отпускает без договорённости | Нет «я отправлю, позвоните когда посмотрите» |
| Кто кому звонит и когда | Явно оговорено |

## Оценка не категоричная

Чек-лист **живой** — руководитель добавляет и убирает критерии, регулирует вес каждого.
Агент оценивает гибко: ситуативное нарушение ≠ системная ошибка.
Системная ошибка = нарушение одного критерия в 3+ звонках подряд.

---

## Оценка самого скрипта

Агент также анализирует эффективность скрипта — не только менеджера:
- Какие этапы вызывают больше всего затруднений (паттерн по менеджерам)
- Какие МП используются редко → возможно не нужны или плохо написаны
- Где менеджеры чаще всего отклоняются от скрипта → скрипт нереалистичен
