## Концепция

Модуль для обучения менеджеров на «цифровых двойниках» реальных клиентов.

**Ключевой принцип:** Агент берёт реальные портреты клиентов из базы диалогов и играет их роль. Менеджер тренирует навык ведения по скрипту. Шестерёнка выявляет пробелы в самом скрипте.

---

## Процесс: создание сценария (руководитель через агент)

```
Руководитель (в admin-агент):
"Создай сценарии для волоконного лазера"

Агент:
1. Поиск диалогов с тегом "волоконный лазер" → нашёл 15
2. Кластеризация по типам клиентов:
   - Технарь (8 диалогов)
   - Производство без опыта (11 диалогов)
   - Перекупщик (4 диалога)
3. Для каждого кластера → черновик досье клиента:
   - Тип клиента, манера речи
   - 3-5 реальных возражений из диалогов
   - Провокационные ситуации
   - Как успешно закрывались похожие звонки
4. Показывает 3 черновика → руководитель правит → тест-прогон → публикация
```

**Важно:** руководитель обязательно делает тест-прогон перед публикацией (5-10 мин) — гарантирует качество сценария.

Досье сценария компактно (~5К токенов) — не все 200 диалогов в памяти, только нужный кластер.

## Процесс: сессия тренировки (менеджер)

```
1. Менеджер открывает тренажёр
2. Выбирает: тип скрипта (Квалификация / Дожим) + сценарий
   ИЛИ видит наряд-задание от руководителя
3. Агент берёт роль клиента
4. Менеджер ведёт диалог
5. Агент:
   - Держит роль клиента (не выходит из образа)
   - В нужный момент вводит провокации: "Я подумаю", "У конкурентов дешевле"
   - При провокации "Я подумаю" → ожидает переключения на скрипт Дожима
6. После сессии: агент выходит из роли → анализ → отчёт
```

**Почему нельзя использовать дешёвую модель для роли клиента:**
Дешёвая модель легко «сломать» — она не заметит технически неверный ответ и продолжит как ни в чём не бывало. Клиент должен реагировать реалистично.

Решение: Claude Haiku + prompt caching (досье клиента кэшируется → дёшево).

## Оценка после сессии

```
Отчёт агента:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Сценарий: Технарь → CO2 лазер 1390
Длительность: 18 реплик

Этап "Открытие":        9/10 ✅
Этап "Квалификация":    6/10 ⚠️
  - Не уточнил объём производства
  - Не использовал МП про акрил 10мм

Этап "Возражения":      4/10 ❌
  - Клиент спросил про 380В — ответ неполный
  - Не отработал возражение по цене чиллера

Провокация "Я подумаю": ✅ переключился на Дожим
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

Агент оценивает каждый этап **отдельно** по функционалу скрипта.

## Два режима работы

### 1. Свободная тренировка
Менеджер сам выбирает сценарий из опубликованных. Тренируется произвольно.

### 2. Наряд-задание (от руководителя)
```
Руководитель назначает: Иван → [Сценарий CO2 Эконом, Сценарий Волокно Технарь] до пятницы
→ Менеджер видит задание в тренажёре
→ Проходит сценарии
→ Агент формирует итоговый отчёт по выполнению наряда
→ Руководитель видит аналитику
```

Сущность в Firestore:
```json
{
  "managerId": "ivan-123",
  "scenarios": ["scenario-co2-econom", "scenario-fiber-tech"],
  "deadline": "2026-05-23",
  "status": "in_progress",
  "completedAt": null,
  "agentReport": null
}
```

## Шестерёнка → пробелы скрипта

```
Менеджер встретил ситуацию без ответа →
нажимает шестерёнку → "Не знал что ответить про заземление 380В"
→ падает руководителю (заметка типа "пробел в скрипте")
→ руководитель через агент добавляет МП
→ следующая тренировка уже с новым знанием
```

Тренажёр — это не только обучение, но и **инструмент выявления пробелов в скрипте**.

## Накопительная аналитика прогресса

> Подробно: [[11_analytics]]

Дашборд по каждому менеджеру:
- Количество тренировок, динамика по неделям
- Средний балл по каждому этапу скрипта
- Системные ошибки (встречаются в N из M сессий)
- Сравнительная статистика по команде (обезличенная)
